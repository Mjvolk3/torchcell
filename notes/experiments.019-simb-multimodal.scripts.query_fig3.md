## 2026.07.22 - WS2 Fig-3 "one embedding, many phenotypes" build + overlap census

Script: `experiments/019-simb-multimodal/scripts/query_fig3.py`
Queries: `experiments/019-simb-multimodal/queries/fig3_build.cql` (full) and
`fig3_core.cql` (default build), generated by `generate_fig3_cql.py`.
Results: `experiments/019-simb-multimodal/results/fig3_overlap_census.json`.

Builds a training-ready `Neo4jCellDataset` (converter=None,
`MeanExperimentDeduplicator`, `GenotypeAggregator`, `SubgraphRepresentation`)
that UNIONs expression (Kemmeren + Sameith Sm/Dm microarray, `node`) + morphology
(Ohya CalMorph, `global`) + single/double fitness (Smf/Dmf, `global`), aggregated
per genotype. Media/temperature are carried (Ohya YPD/25C, others SC/SGA-sel
30C/26C) and never filtered. Fitness blocks require ALL perturbations to be
deletions so a fitness record only lands on the SAME deletion strain (not a
ts-allele / DAmP of the same gene).

### Live-DB overlap (deletion-filtered) — genotypes that COULD carry >=2 modalities

- Q1 — Sameith Dm double-KO expression pairs (72) with a matching double-mutant
  fitness pair: **57** (all 57 via DmfCostanzo2016; +8 via DmfKuzmin2020, 0 via
  DmfKuzmin2018).
- Q2 — single-KO genotypes with BOTH expression and single-mutant fitness:
  **1360** (Kemmeren∩Smf 1360, Sameith-Sm∩Smf 79).
- Q3 (predicted from gene-set overlap) — **4425** unique genotypes carry >=2
  modalities: fitness+morphology 2894, expression+fitness+morphology 1326,
  expression+morphology 114, expression+fitness 91. Headline
  **expression+fitness = 1417**, expression+morphology = 1440.

The DB overlap is HEALTHY — the Fig-3 thesis has ~1.4k genotypes with both an
expensive expression vector and a cheap fitness scalar.

### BLOCKER 1 — the aggregator only REALIZES a fraction of that overlap

After the real build, the aggregation LMDB has **9407** groups; only **1448**
carry >=2 modalities, and the headline **expression+fitness collapses to 90**
(expression+morphology 1358; ZERO fitness+morphology and ZERO triples form).

Mechanism (confirmed in code): `MeanExperimentDeduplicator.process`
(`deduplicate.py` L108-116) rewrites only DUPLICATED genotypes to
`MeanDeletionPerturbation` (`perturbation_type="mean_deletion"`); SINGLETON
genotypes keep their original marker-specific type (`kanmx_deletion`,
`sga_kanmx_deletion`, ...). `GenotypeAggregator.aggregate_check`
(`genotype_aggregate.py` L22-28) keys on `(systematic_gene_name,
perturbation_type)`. Expression (Kemmeren) and morphology (Ohya) are singletons
that BOTH keep `kanmx_deletion`, so they merge (1358). But fitness is heavily
replicated → `mean_deletion`, so it never matches the expression/morphology
marker key → fitness fails to join. The 90 expr+fitness that do merge are the
cases where the fitness genotype was itself a singleton keeping a marker type
shared with the expression record (e.g. Sameith-Dm sga + DmfCostanzo singleton
sga pairs).

Fix for WS11a: make `GenotypeAggregator` key on the SORTED gene-name set only
(matching `MeanExperimentDeduplicator.duplicate_check`, which already ignores
`perturbation_type`), OR normalise all deletion `perturbation_type`s to a single
canonical value before aggregation. Either unifies the ~1417 expr+fitness /
~1326 triples the DB actually contains.

### BLOCKER 2 — SubgraphRepresentation cannot tensor-ize vector phenotypes

`SubgraphRepresentation._add_phenotype_data` (`graph_processor.py` L529) does
`torch.tensor(all_values, dtype=float)`; expression/CalMorph phenotype fields are
dict-valued (`expression_log2_ratio`, `calmorph`), so vector-phenotype groups
raise `TypeError: must be real number, not dict`. `dataset[idx]` therefore only
builds for SCALAR (fitness/gene-interaction) groups. The runner scans for the
first valid index: **index 2** builds a valid `HeteroData` (indices 0-1 are
vector-blocked) — gene x `[6606, 1600]`, metabolism reaction/metabolite stores,
scalar `phenotype_values`. A vector-capable graph processor is needed before the
expression/morphology groups can be consumed by a model.

### Build note

`fig3_core.cql` (default) drops `DmfKuzmin2018/2020`: together ~1.04M
double-mutant fitness records that add 0 expression-overlapping genotypes (only 8
pairs, already covered). `fig3_build.cql` keeps them for completeness. Run the
census-only path (`FIG3_CENSUS_ONLY=1`) to reproduce the counts without a build.
